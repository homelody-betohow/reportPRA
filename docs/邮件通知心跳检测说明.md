# 邮件通知心跳检测说明（`scripts/jobs/heartbeat.py`）

本说明用于帮助你在本项目中启用并验证「定时任务 + 邮件通知」是否正常工作。

---

## 1. 它做了什么？

- 容器内 `cron` 会按 `docker/python/crontab` 的配置 **每天早上 8:00** 执行一次：
  - `/usr/local/bin/run-heartbeat`（内部会执行 `python /app/scripts/jobs/heartbeat.py`）
- 脚本会尝试通过 SMTP 发送一封通知邮件，并把结果输出到容器日志：
  - 成功：`[cron] 邮件已发送...`
  - 失败：`[cron] 邮件发送失败...`
  - 缺配置：`[cron] 邮件未发送：缺少必要配置...`

---

## 2. 需要配置哪些环境变量？

这些变量写在 `.env`（参考 `.env.example`），并由 `docker-compose.yml` 传入 `python` 容器。

### 2.1 必填（缺任何一个都不会发送）

- `SMTP_HOST`：SMTP 服务器地址（例：`smtp.qq.com` / `smtp.gmail.com` / 企业邮箱 SMTP）
- `MAIL_FROM`：发件人邮箱地址（通常等于 `SMTP_USER`）
- `MAIL_TO`：收件人（支持多个，用英文逗号分隔）

### 2.2 常用（通常也需要）

- `SMTP_PORT`：常见 `587`（STARTTLS）或 `465`（SSL）
- `SMTP_USER`：SMTP 登录账号（一般是邮箱地址）
- `SMTP_PASS`：SMTP 密码/授权码（注意很多邮箱不是登录密码，而是“SMTP 授权码”）
- `SMTP_STARTTLS`：`true/false`（默认 `true`）
- `SMTP_SSL`：`true/false`（默认 `false`）
- `MAIL_SUBJECT`：邮件标题（可选）

### 2.3 163 邮箱推荐配置（优先用这个）

你遇到的 `SMTPServerDisconnected: Connection unexpectedly closed` 在 163 上常见原因是 **端口/TLS 组合不匹配**。
建议使用 **465 + SSL**（更稳定）：

```env
SMTP_HOST=smtp.163.com
SMTP_PORT=465
SMTP_SSL=true
SMTP_STARTTLS=false
SMTP_USER=你的163邮箱@163.com
SMTP_PASS=你的「客户端授权码」

MAIL_FROM=你的163邮箱@163.com
MAIL_TO=收件人1@xx.com,收件人2@xx.com
```

## 3. 本地如何测试脚本是否能运行？

### 3.1 启动（或重建）python 容器

在项目根目录执行：

```bash
docker compose up -d --build python
```

### 3.2 手动执行一次脚本（最快）

```bash
docker compose exec python python /app/scripts/jobs/heartbeat.py
```

你会看到三种结果之一：

- 缺配置：提示你缺少 `SMTP_HOST`、`MAIL_FROM`、`MAIL_TO`
- 发送成功：会输出 `邮件已发送`
- 发送失败：会输出失败原因（例如认证失败、超时、TLS/SSL 不匹配等）

## 4. 本地如何确认“定时任务（cron）”在正常触发？

### 4.1 看 cron 是否已启动

```bash
docker compose exec python bash -lc "ps -ef | grep -E 'cron($| )' | grep -v grep"
```

### 4.2 看定时任务文件是否存在且内容正确

```bash
docker compose exec python bash -lc "ls -l /etc/cron.d/rpa-task && sed -n '1,200p' /etc/cron.d/rpa-task"
```

里面应该有一行类似（示例为每分钟跑一次）：

```cron
0 8 * * * root /usr/local/bin/run-heartbeat >> /proc/1/fd/1 2>> /proc/1/fd/2
```

### 4.3 观察容器日志（等到下一个整分钟）

```bash
docker compose logs -f python
```

如果 cron 正常，每天 8:00 你都会看到脚本输出（成功/失败都会输出一行）。

## 5. 常见问题排查

### 5.1 `邮件未发送：缺少必要配置...`

说明 `.env` 里还没有填：`SMTP_HOST`、`MAIL_FROM`、`MAIL_TO`（至少这三项要有）。

### 5.2 认证失败（用户名/密码错误）

很多邮箱（尤其 QQ/163/企业邮箱）需要在邮箱后台开启 SMTP，并生成“授权码”作为 `SMTP_PASS`，不是网页登录密码。

### 5.3 TLS/SSL 选错导致握手失败

常见组合：

- `SMTP_PORT=587` + `SMTP_STARTTLS=true` + `SMTP_SSL=false`（最常见）
- `SMTP_PORT=465` + `SMTP_SSL=true`（通常不需要 STARTTLS）

如果报错里有 `SSL` / `STARTTLS` / `wrong version number`，优先检查这三项。

### 5.4 其他机器（Windows）部署一直报 `bash\r` 相关错误

这通常是脚本被 Windows 自动换成 CRLF 结尾导致。仓库已通过：

- `.gitattributes` 强制 `docker/python/*.sh` 与 `docker/python/crontab` 使用 LF
- `docker/python/Dockerfile` 构建时 `sed -i 's/\r$//' ...` 双保险

若仍遇到旧缓存镜像问题，建议：

```bash
docker compose build --no-cache python
docker compose up -d python
```
